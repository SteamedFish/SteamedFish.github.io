事务的一致性模型

2019-06-13

文章目录

  • 传统的一致性模型
    • Read Uncommitted
    • Read Committed
    • Monotonic Atomic View

传统的事务一致性模型，使用隔离级别来实现。传统的隔离级别，就是我们所熟悉的四种：

  • Read Uncommitted

  • Read Committed

  • Repeatable Reads

  • Serializable

但是，在实际的实现过程中，各个数据库实现了一些中间级别的一致性模型（例如 MySQL 的 Repeatable Reads 隔离级别，实际上比真正的 Repeatable Reads 要弱一点，在某项特殊 情况下并不能满足 Repeatable Reads 的要求）。所以实际上真实情况会比较复杂。

当引入了分布式系统之后，情况就更加复杂了。分布式系统除了面临各种传统系统的一致性 要求之外，还需要面临更多的挑战，因此诞生了更多的一致性模型。

传统的一致性模型

Read Uncommitted

这是一个事务模型。按照字面意义，一个事务允许读到另外一个事务尚未提交的修改（脏读）。

  • 允许脏读，也就是一个事务允许读到另外一个事务尚未提交的修改。

  • 不允许脏写，也就是一个事务不允许覆盖另外一个事务尚未提交的写入。

  • 在分布式数据库下，它可以保证完全的可用性，任何形式的网络故障或者节点故障情况下， 所有非故障节点都可以正常工作而依然保证该一致性。

Read Committed

这是一个事务模型。按照字面意义，一个事务能读到其他事务已经提交的修改（不允许脏 读）。

  • 不允许脏读

  • 一个事务如果能读取到其他事务提交的修改，则会出现不可重复读，也就是一个事务反复 读取同一个数据，其读取结果可能不一致（已经被其他事务提交了修改）。

  • 在分布式数据库下，它可以保证完全的可用性，任何形式的网络故障或者节点故障情况下， 所有非故障节点都可以正常工作而依然保证该一致性。

  • 在分布式数据库下，它不保证严格的实时性。一个事务不保证一定能立刻获取到另外一 个事务的写入结果。

  • 它也不保证单个进程下的事务顺序。一个进程可能在一个事务中获取到了另外一个事务 的写入结果，但是该进程的下一个事务可能又无法获取同一个写入结果。

Monotonic Atomic View

这是一个事务模型，字面意义是单调原子视图。

在这个模型下，事务可以涉及多个按顺序执行的简单操作，并且可以操作多个对象。

  • 不允许中间读（Intermediate Reads）。
